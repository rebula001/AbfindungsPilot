# 05 - i18n

> **In 30 Sekunden:** `vue-i18n` 11 mit zwei Locales (DE Default, ZH). `de.ts` und `zh.ts` müssen **strukturell 1:1** identisch sein (vue-i18n erzwingt das via TypeScript). Vor jedem Commit: `npm run i18n:check`.

## Konfiguration

[`src/i18n/index.ts`](../../../frontend/src/i18n/index.ts):

```ts
export const i18n = createI18n({
  legacy: false,
  locale: loadInitialLanguage(), // localStorage `abfindungspilot.language.v1`, sonst `de`
  fallbackLocale: 'de', // unübersetzte ZH-Keys -> DE-Anzeige
  warnHtmlMessage: false, // Tooltips enthalten <p>/<ul>/<strong>
  messages: { de, zh }
});
```

`fallbackLocale: 'de'` bedeutet: ein fehlender ZH-Key zeigt den deutschen Wert statt einer kryptischen `t('xxx')`-Fehlermeldung. **Das ist trotzdem ein Bug** und wird vom Check-Skript reportet.

## Strukturregel - DE und ZH sind isomorph

Wegen TypeScript-Typinferenz von vue-i18n erzwingt `vue-tsc -b` (in `npm run build`), dass `de` und `zh` denselben Schlüsselbaum haben. Eine Diskrepanz -> Build-Fehler.

```ts
// ❌ Bricht den Build:
// de.ts: { foo: { bar: 'X' } }
// zh.ts: { foo: { baz: 'Y' } }    // bar fehlt, baz unerwartet

// ✅ Korrekt:
// de.ts: { foo: { bar: 'X' } }
// zh.ts: { foo: { bar: 'Y' } }
```

## Übersetzungsregel - deutsche Steuerterminologie behalten

In der chinesischen Übersetzung **dürfen die deutschen Steuer-Begriffe nicht eingedeutscht werden**, weil:

1. Sie sind Fachbegriffe ohne präzise chinesische Entsprechung
2. Der User wird sie ohnehin in Steuerbescheid / Steuerberatung wiedersehen
3. Das App-Publikum sind in Deutschland lebende Chinesischsprechende

| DE | ZH |
| --- | --- |
| `Abfindung` | `Abfindung` (遣散费) - (erstmalige Erwähnung), danach `Abfindung` |
| `KFB` | `KFB` (儿童免税额) - (1x), danach `KFB` |
| `Soli` | `Soli` (团结附加税) - (1x), danach `Soli` |
| `Fünftelregelung` | `Fünftelregelung` (五分之一规则) - (1x), danach `Fünftelregelung` |
| `Progressionsvorbehalt` | `Progressionsvorbehalt` (累进保留) - (1x), danach `Progressionsvorbehalt` |
| `ALG I` | `ALG I` (失业救济金) - (1x) |
| `GKV` / `PKV` | `GKV` (法定医保) / `PKV` (私人医保) |
| `zvE` | `zvE` (应税所得) |
| `Veranlagungsart` | `Veranlagungsart` (申报方式) |
| `Splittingtarif` | `Splittingtarif` (分摊税率) |
| `§ 32a EStG`, `§ 158 SGB III` | unverändert |

Bundesland-Namen bleiben deutsch (`Bayern`, `Baden-Württemberg`).

## Mehrzeilige Inhalte: `tList()` und `tm()`

Für aufzählungsartige Inhalte (Erklärungstexte, Formel-Schritte) wird im Locale-File ein Array gespeichert und im Template über `tm()` iteriert. Beispiel siehe `CalculationView.vue`.

```ts
// de.ts
calculation: {
  fuenftelExplanation: ['Schritt 1: ...', 'Schritt 2: ...', 'Schritt 3: ...']
}
```

```vue
<li v-for="(line, i) in tm('calculation.fuenftelExplanation')" :key="i">{{ line }}</li>
```

Vorteile: pro Zeile übersetzbar, leichter zu pflegen als ein riesiger HTML-String.

## HTML-Tooltips

Manche Tooltips brauchen Formatierung (`<p>`, `<ul>`, `<strong>`). Das ist erlaubt, weil:

- Inhalte stammen ausschließlich aus statischen Locale-Files (kein User-Input -> kein XSS-Risiko)
- `warnHtmlMessage: false` unterdrückt die intlify-Warnung

Eingebunden via `v-tooltip` mit `escape: false`:

```vue
<button
  type="button"
  class="pi pi-info-circle ..."
  v-tooltip.right="{
    value: t('form.taxClassTooltip'),
    escape: false,
    class: 'tax-class-tooltip',
    showDelay: 150
  }"
  :aria-label="t('form.taxClassTooltipAria')"
/>
```

## Sprachumschalter

Toolbar oben rechts: zwei Buttons DE / ZH. Klick -> `cacheLanguage(lang)` schreibt localStorage und setzt `i18n.locale.value`.

Persistenz-Key: `abfindungspilot.language.v1` (KEIN TTL, anders als der Snapshot-Key).

## Check-Skript

[`scripts/check-i18n-keys.mjs`](../../../frontend/scripts/check-i18n-keys.mjs) prüft drei Dinge:

| Befund | Bedeutung | Aktion |
| --- | --- | --- |
| **Missing** | Source-Code referenziert Key, der in `de.ts` oder `zh.ts` fehlt | hinzufügen oder Tippfehler korrigieren |
| **Cross-locale diff** | Key in DE vorhanden, in ZH fehlt (oder umgekehrt) | nach-übersetzen |
| **Unused** | Key in Locale, aber nirgends im Source verwendet | Kandidat zum Löschen - manuell prüfen (statisch nicht analysierbare Keys werden ignoriert) |

Erkennungsstrategie (vereinfacht):

- `` `a.b.c` `` / `"a.b.c"` / `'a.b.c'` direkt -> verwendet
- `` `a.b.` + var + `.b` `` markiert
- `` "a.b.${x}.c" `` -> Keys mit Präfix `a.b.` und Suffix `.c` markiert
- `t('a.b')` -> `a.b` und alle Children markiert
- `t('a.b.c.d')` -> alle Eltern (`a.b`, `a.b.c`) auch markiert

**Exit-Code:** Missing oder Cross-locale-Diff > 0 -> `1` (CI-blockierend). Unused -> nur Report, kein Fail.

## Workflow für neue UI-Texte

1. Key in `de.ts` an passender Stelle hinzufügen
2. Selben Key in `zh.ts` mit Übersetzung hinzufügen (deutsche Steuer-Begriffe behalten!)
3. Im Template / TS via `t('xxx')` referenzieren
4. `npm run i18n:check` -> muss grün sein
5. `npm run build` -> `vue-tsc` bestätigt strukturelle Konsistenz
